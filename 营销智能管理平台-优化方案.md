# 营销智能管理平台 — 优化技术方案

## 一、项目概述

### 1.1 一句话定位

面向空间大数据事业部管理者的**营销情报中枢**——对内结构化存档日报数据并生成汇报页面，对外自动采集行业情报并以精美长图通过钉钉推送全员。

### 1.2 核心用户

| 角色 | 使用方式 | 备注 |
|------|---------|------|
| **管理者** | 登录平台操作：查看日报存档、生成报告、浏览资讯、管理速递 | 唯一操作者 |
| **领导** | 通过钉钉收到日报/周报页面链接，打开浏览 | 被动接收 |
| **一线人员**（20-50人） | 通过钉钉机器人提交日报 + 收到每日速递长图 | 不直接使用平台 |

### 1.3 系统协作流程

```
一线人员 ──(日报)──> 钉钉机器人 ──(JSON文件)──> 平台解析入库 ──> 管理者查看/检索
                                                                │
                                                  生成日报/周报页面 ──> 钉钉链接推送给领导

云端爬虫 ──(定时采集)──> 云端数据库 ──(API同步)──> 内网平台数据库
                                                        │
                                          ┌──────────────┼──────────────┐
                                          ▼              ▼              ▼
                                    资讯中心展示    每日速递自动生成    管理者筛选审阅
                                                        │
                                          渲染HTML ──> 截图为长图 ──> 钉钉推送全员
```

## 二、功能模块设计

### 模块一：日报/周报存档与报告生成

#### 2.1.1 数据解析与入库

**解析流水线设计（三级容错）：**

```
JSON文件 → 格式嗅探 → 直接解析(成功率高时) 
                     → LLM标准化(格式异常时) → 结构化数据 → 入库
                     → 人工标记(极端异常)
```

- **嗅探层**：自动识别 JSON 文件结构类型（多种模板兼容）
- **规则解析**：对已知稳定格式，直接用规则提取，不调 LLM（省成本）
- **LLM 兜底**：格式偏差大时调用 Qwen-Flash（0.15元/百万Token）做标准化
- **原始归档**：所有原始 JSON 文件一律保留，支持回溯和重新解析

**标准化输出 Schema：**

```json
{
  "person_name": "张伟",
  "date": "2026-06-10",
  "role": "销售",
  "department": "华北组",
  "activities": [
    {
      "type": "拜访客户",
      "target": "北京市公安局",
      "opportunity": "智慧警务地理信息平台",
      "description": "与信息中心王主任沟通三期建设需求",
      "confidence": 0.95
    }
  ]
}
```

> `confidence` 字段记录 LLM 标注置信度，低于阈值时在管理端标记为待审核。

#### 2.1.2 行为标签体系

| 行为类别 | 说明 | 色标 |
|---------|------|------|
| 拜访客户 | 上门/远程客户接触 | 蓝色 |
| 商机跟进 | 商机发现、推进、报价、谈判 | 橙色 |
| 方案撰写 | 技术方案、投标方案、POC | 青色 |
| 项目推进 | 已签约项目的交付、验收 | 绿色 |
| 渠道拓展 | 合作伙伴开发、渠道维护 | 紫色 |
| 回款跟进 | 催款、对账、开票 | 红色 |
| 内部协作 | 内部会议、跨部门协调 | 灰色 |

- LLM 自动标注 + 管理者可修正
- 修正结果自动记录为 few-shot 示例，持续优化后续标注准确率

#### 2.1.3 多维度查看与检索

管理者可自由组合以下维度筛选：

- **按人员**：查看某人某时段所有事项，支持点击展开个人详情
- **按组织**：华北组/华东组/行业组整体工作情况
- **按工种**：销售 vs 解决方案对比
- **按行为类型**：如只看所有"拜访客户"记录
- **按商机**：某个商机的所有相关活动聚合展示
- **按日期范围**：任意时间段回溯
- **全文搜索**：描述内容、客户名称、商机名称关键词检索

#### 2.1.4 日报/周报页面生成

- 基于当天/当周存档数据，一键生成精美 HTML 页面
- 页面支持按岗位、组织筛选，领导打开后可自行切换视角
- 支持日期前后翻看历史
- 管理者可添加备注后生成页面链接
- 链接通过钉钉机器人推送给领导

#### 2.1.5 个人工作详情

点击任何人员展开详情面板：
- 近期所有工作事项列表
- 活跃天数、拜访次数、商机跟进次数等关键指标
- 为 1v1 沟通、绩效评估提供数据支撑

#### 2.1.6 数据导出

所有存档数据支持按筛选条件导出 Excel。

### 模块二：资讯中心与每日速递

#### 2.2.1 资讯中心（Tab 2）

爬虫采集数据的集中展示窗口，按四个板块分类浏览：

| 板块 | 数据来源 | 更新频率 |
|------|---------|---------|
| 标讯信息 | 政府采购网、公共资源交易中心等 | 每天 2 次 |
| 市场动态 | 政数局/住建部官网、分析机构报告 | 每天 1 次 |
| 竞对监控 | 竞对官网、中标信息、招聘动态 | 每天 1 次 |
| AI/Agent | 科技媒体、GitHub Trending | 每天 1 次 |

管理者在此浏览，不做编辑发布——发布由每日速递自动完成。

#### 2.2.2 爬虫采集详细方案

> 详见下方「附录A：爬虫模块详细设计」

#### 2.2.3 每日速递自动生成

**执行流程（每天 10:00 自动触发）：**

```
1. 汇总当天所有爬虫数据
2. 按板块分组（有数据才生成对应板块）
3. LLM 生成每条数据的一句话摘要
4. 套用精美 HTML 模板渲染页面
5. Playwright 截图生成高清长图（PNG）
6. LLM 生成一段导语文字（如"今日速递：新增5条标讯..."）
7. 调用钉钉机器人 API：
   a. 上传长图到钉钉获取 MediaId
   b. 发送 Markdown 消息（导语文字 + 图片）
```

**速递页面板块设计：**
- 标讯板块：卡片式展示，含项目名、预算、采购单位、截止日期、匹配度
- 市场动态板块：按主题分组，每条配摘要
- 竞对板块：按竞争对手分组，配竞对标识
- AI/Agent 板块：按分类（大模型/Agent/工具），配一句话摘要

#### 2.2.4 速递历史存档

所有已生成的速递按时间排列存档，管理者可查看推送历史和各板块数据量。

## 三、技术架构（互联网+内网混合部署）

### 3.1 网络拓扑与数据流全景

**核心架构原则：云端为主、内网为辅、单向数据流，两平台合并部署**

| 原则 | 说明 |
|------|------|
| 云端承载所有业务逻辑 | 营销平台 + 经营工作台 + 爬虫，全部在同一台云服务器 |
| 内网只做一件事 | 机器人收集日报 → 生成JSON → 上传到云端 |
| 数据流单向 | 内网主动PUSH到云端（内网出站访问互联网，无需反向穿透） |
| 两平台共用数据库 | 营销平台和经营工作台共享同一PostgreSQL实例，通过统一ID直接关联 |
| 钉钉充当消息总线 | 云端直接调用钉钉API推送消息，不经过内网 |

### 3.2 营销平台与经营工作台的合并方案

#### 合并策略：同服务器、同PostgreSQL实例、两个 Schema

```
PostgreSQL 实例
├── Schema: marketing     (营销平台数据)
│   ├── staff               人员
│   ├── activities          日报活动记录
│   ├── crawler_items       爬虫数据
│   ├── daily_express       每日速递
│   └── report_pages        报告页面
│
└── Schema: business      (经营工作台数据，已有)
    ├── opportunities       商机表 (opportunity_id, 商机名, 金额, 阶段...)
    ├── customers           客户表 (customer_id, 客户名, 行业...)
    ├── contracts           合同表
    └── ...                 其他经营数据
```

#### 数据关联方式

营销平台的 `activities` 表中的商机字段，通过统一 ID 直接关联经营工作台的数据：

```sql
-- 营销平台查看某商机的经营数据 (跨Schema查询)
SELECT 
    a.description,
    a.activity_type,
    o.opportunity_name,
    o.amount,
    o.stage,
    o.expected_close_date,
    c.customer_name,
    c.industry
FROM marketing.activities a
LEFT JOIN business.opportunities o ON a.opportunity_id = o.opportunity_id
LEFT JOIN business.customers c ON o.customer_id = c.customer_id
WHERE a.report_date = '2026-06-10';
```

#### 合并展示效果

**在营销平台中，商机相关数据会自动富化：**

| 场景 | 展示内容 | 数据来源 |
|------|---------|----------|
| 日报“按商机”视图 | 商机名称 + 金额 + 当前阶段 + 预计签约时间 | 经营工作台 |
| 点击商机名 | 弹出商机详情卡片（金额、客户、阶段、负责人、历史动态） | 经营+营销合并 |
| 日报活动卡片 | 活动描述下方显示关联商机的金额和状态 | 经营工作台 |
| 周报统计 | 本周跟进商机总金额、按阶段分布 | 经营工作台 |
| 个人详情 | 负责的商机列表及各商机状态 | 经营工作台 |

**核心优势：营销平台只读经营数据，不写入，不影响经营工作台的正常运行。**

### 3.3 五大数据流详解

#### 流①：日报数据从内网到云端

```
员工私聊机器人(钉钉) → 钉钉云转发 → 内网机器人后台
                                          │
                                    私有化LLM处理对话
                                    结构化提取日报内容
                                          │
                                    生成JSON文件
                                          │
                                    同步脚本 (15min一次)
                                          │
                              HTTPS POST → 云端平台 /api/import/json
                                          │
                                    云端解析入库
```

**内网同步脚本（部署在内网机器人服务器上，~50行代码）：**

```python
import httpx, os, json

CLOUD_API = "https://marketing.your-domain.com/api/import/json"
API_KEY = "your-secret-api-key"
JSON_DIR = "/data/bot/daily_reports/"
SYNCED_LOG = "/data/bot/.synced_files"

def sync_new_files():
    synced = load_synced_log()
    for f in os.listdir(JSON_DIR):
        if f.endswith('.json') and f not in synced:
            with open(os.path.join(JSON_DIR, f)) as fp:
                data = json.load(fp)
            resp = httpx.post(CLOUD_API,
                json={"filename": f, "content": data},
                headers={"Authorization": f"Bearer {API_KEY}"},
                timeout=30)
            if resp.status_code == 200:
                synced.add(f)
    save_synced_log(synced)

# 内网cron: */15 * * * * python /opt/sync_to_cloud.py
```

#### 流②：云端直接推送钉钉消息（不经过内网）

```
云端平台 ─── 直接调用钉钉 Webhook API ───> 钉钉云服务 ───> 员工收到消息
```

**重要说明**：钉钉本身是互联网服务，云端平台只需配置同一个群机器人的 Webhook URL，就能直接发消息给员工。这与内网机器人后台互不干涉。

推送场景：
- 每日速递（文字+长图）→ 全员
- 日报/周报页面链接 → 领导
- 爬虫异常告警 → 管理者

#### 流③：管理者/领导访问平台

```
管理者/领导 ── HTTPS ──> 云端平台 (https://marketing.your-domain.com)
                                    │
                              Nginx + SSL证书
                                    │
                         管理者：账号密码登录，全功能权限
                         领导：报告链接+Token，无需登录，有30天有效期
                         员工：不访问平台，只看钉钉推送
```

#### 流④：爬虫采集（全在云端完成）

```
互联网网站 ──(爬取)──> 云端爬虫引擎 ──> 同一个 PostgreSQL ──> 平台直接读取
```

爬虫和平台都在云端、共用一个数据库，**无需任何数据同步**，极大简化架构。

#### 流⑤：LLM 分工

```
内网私有化LLM：                    云端通义千问API：
│                                │
├─ 机器人对话理解              ├─ 日报JSON二次结构化解析
├─ 员工日报内容提取          ├─ 爬虫数据相关度评分
└─ 日报初步结构化             ├─ 每日速递摘要生成
   (已有能力，不需要改)       ├─ 周/月报AI总结
                                └─ 行为标签自动分类

互不依赖，各干各的。
```

### 3.4 技术选型

| 层次 | 技术 | 部署位置 | 理由 |

| **定时任务** | APScheduler | 云端 | 爬虫+速递+状态检查 |
| **页面截图** | Playwright | 云端 | HTML渲染为高清PNG |
| **云端LLM** | 通义千问 Qwen-Flash/Doc-Turbo | 云端 | Flash成本极低，Doc-Turbo精度高 |
| **内网LLM** | 私有化部署模型 | 内网 | 已有，机器人使用 |
| **消息推送** | 钉钉 Webhook | 云端调用 | 直接调钉钉API，不经内网 |
| **反向代理** | Nginx + Let's Encrypt | 云端 | HTTPS + 前端托管 |
| **数据上传** | Python同步脚本 + Cron | 内网 | 轻量级，定时push JSON |
| **部署** | Docker Compose | 云端 | 一键启动所有服务 |

### 3.5 安全设计

| 安全层 | 措施 | 说明 |
|---------|------|------|
| **平台访问** | 账号密码 + JWT Token | 管理者登录，Token 24h有效期 |
| **报告页面** | 一次性Token链接 | 领导点开无需登录，30天过期 |
| **数据上传接口** | API Key + IP白名单 | 只允许内网出口IP调用 |
| **钉钉Webhook** | HMAC-SHA256签名 | 防止伪造请求 |
| **数据库** | 不暴露公网端口 | 只允本机/Docker内网访问 |
| **HTTPS** | Let's Encrypt免费证书 | 所有传输加密 |

### 3.6 钉钉推送方案

**图片消息发送流程（云端直接推送）：**

```
1. Playwright 截图生成 PNG 长图
2. 调用钉钉上传接口 → 获取 media_id
3. 构造 Markdown 消息：
   { "msgtype": "markdown", "markdown": {
       "title": "每日速递",
       "text": "## 6月10日 每日速递\n..."
   }}
4. POST 到 Webhook URL（带HMAC签名验证）
```

### 3.7 LLM 调用策略

| 场景 | 模型 | 位置 | 成本 |

### A.2 标讯爬取——剑鱼网 RPA 方案

#### 核心思路

剑鱼网有 VIP 账号但无 API，采用 **Playwright + AI 操控浏览器** 的 RPA 方案模拟真人操作。

#### 渐进式实施路线

```
第一阶段（半自动）：
  管理者在平台点击“触发爬取” → 系统自动执行浏览器操作 → 完成后通知
  遇到验证码/异常时，通知管理者人工干预

第二阶段（全自动）：
  定时任务自动触发（每天 8:00 + 14:00）
  对常见验证码自动处理（滑块、图片识别）
  异常时自动重试 + 钉钉告警
```

#### 技术实现详解

**流程设计：**

```python
# 剑鱼网爬取流程伪代码
async def crawl_jianyu():
    # 1. 启动浏览器（复用Session避免频繁登录）
    browser = await playwright.chromium.launch(headless=True)
    context = await load_or_create_session()  # Cookie持久化
    
    # 2. 检查登录状态
    if not await is_logged_in(context):
        await login_with_credentials(context)  # 账号密码登录
        await handle_captcha_if_needed(context)  # 验证码处理
    
    # 3. 导航到标讯搜索页
    await navigate_to_search(context)
    
    # 4. 输入关键词搜索（遍历关键词库）
    for keyword in KEYWORD_LIST:
        await search_keyword(context, keyword)
        
        # 5. 解析搜索结果列表
        items = await parse_search_results(context)
        
        # 6. 进入详情页获取完整信息
        for item in items:
            detail = await get_item_detail(context, item.url)
            await save_to_db(detail)
        
        # 7. 随机延迟（模拟人工操作节奏）
        await random_delay(2, 5)  # 2-5秒随机等待
    
    # 8. 保存Session供下次复用
    await save_session(context)
```

**关键词库（可配置）：**

```yaml
bidding_keywords:
  core:  # 核心关键词，每次必爬
    - "地理信息"
    - "GIS"
    - "实景三维"
    - "CIM"
    - "数字孪生"
    - "智慧城市"
    - "空间大数据"
  extended:  # 扩展关键词，轮换爬取
    - "地址服务"
    - "三维建模"
    - "地图服务"
    - "地名地址"
    - "时空大数据"
    - "城市信息模型"
    - "自然资源信息化"
    - "不动产登记系统"
```

**反检测策略：**

| 策略 | 实现方式 | 说明 |
|------|---------|------|
| Cookie持久化 | 保存浏览器Session到本地文件 | 避免每次重新登录 |
| 随机延迟 | 每次操作间隔2-5秒随机等待 | 模拟人工浏览节奏 |
| 模拟鼠标移动 | Playwright的mouse.move模拟真实轨迹 | 防止行为检测 |
| UA轮换 | 每次启动随机选择User-Agent | 降低指纹识别风险 |
| 频率控制 | 每天最多爬取2次，间隔6小时+ | 模拟正常VIP用户使用频率 |
| IP保护 | 内网测试时用本机IP，上云后换IP | 单IP低频足够安全 |

**验证码处理策略：**

```
第一阶段（半自动）：
  → 检测到验证码 → 截图保存 → 钉钉通知管理者 → 等待人工处理
  → 管理者远程登录服务器手动处理验证码
  → 或在平台上查看验证码图片并输入

  → 滑块验证：Playwright模拟拖拽（常见滑块成功率 > 90%）
  → 图片验证码：接入第三方OCR服务（如百度OCR）

### A.6 数据处理流水线

#### 爬取后的统一处理流程

```
原始数据 → 去重判断 → 相关度评分 → 摘要生成 → 入库
         │            │             │
    URL+标题哈希   关键词匹配     Qwen-Flash
    已存在则跳过   +LLM打分      生成一句话摘要
                   <60分丢弃
```

#### 去重策略

```python
def is_duplicate(item):
    # 方法1：URL完全匹配
    if db.exists(source_url=item.url):
        return True
    # 方法2：标题相似度 > 0.85 且同一天
    similar = db.find_similar_title(item.title, threshold=0.85, same_day=True)
    if similar:
        return True
    return False
```

#### 相关度评分策略

```python
def calculate_relevance(item, category):
    score = 0
    
    # 关键词匹配得分 (0-60分)
    matched = [kw for kw in KEYWORDS[category] if kw in item.title + item.content]
    score += min(len(matched) * 15, 60)
    
    # LLM 评分 (0-40分)，仅当关键词得分 > 20 时才调LLM（省成本）
    if score > 20:
        llm_score = await llm_rate_relevance(item, category)
        score += llm_score * 0.4
    
    return min(score, 100)
```
