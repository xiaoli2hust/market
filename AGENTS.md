# Market 数据采集中心工程协作约束

本文件是当前 `market` 项目的最高工程协作约束。所有 Agent、开发者和代码审查者进入本项目后，必须先读本文件，再读 `README.md`、后端说明、前端说明和相关业务文档。任何页面、接口、Agent、数据表、采集源、提示词、配置项和验收脚本，都不能违背本文件。

本项目不是传统 CRM，不是固定 BI 看板，也不是把数据库查询页面套一层 AI 外壳。它面向 G 端销售与市场团队，目标是形成一条可追溯、可运营、可复盘的增长闭环：

```text
外部情报 -> 线索识别 -> 销售认领 -> 机器人追问 -> 签单预测 -> 回款预测 -> 管理复盘
```

如果历史文档、旧页面、旧代码或旧交接记录与本文件冲突，优先级为：用户最新明确要求 > 本文件 > `README.md` > 当前代码事实 > 历史文档。

## 1. 项目模块边界

当前产品分为四个一级模块：

1. 日报周报：服务管理者掌握团队每天和每周在做什么，识别关键动作、风险事项、外部情报变化和需要跟进的管理动作。
2. 市场洞察：承载外部情报采集、分析和行动建议，包括标讯雷达 Agent、政策与市场跟踪 Agent、市场数据采集中心、竞对监控和行业动态知识补充。
3. 商机中心：承载线索转商机、销售认领、机器人追问、商机进展跟踪、签单预测和回款预测。Agent 只负责跟进、提醒、归因和风险提示，不替销售编造预测。
4. 管理中心：承载采集源、关键词、模型、提示词、账号、权限、API Key、钉钉、调度和系统配置。管理中心不是展示页，配置必须真实影响运行链路。

新增或调整模块前，必须先说明它属于哪条业务链路、解决谁的问题、输入是什么、输出是什么、结果在哪里被使用。

## 2. 基础契约前置

新增页面、接口、Agent、数据表、导入流程、采集源、分析报表或后台配置前，必须先确认对应业务契约：

1. 本体定义：对象是什么，是否已有对象可以承载。
2. 字段字典：字段含义、单位、格式、是否必填、是否可为空。
3. 状态枚举：状态值、流转方向、终态、异常态。
4. 关系口径：对象之间如何关联，关联是否可追溯。
5. 事件类型：什么变化会形成事件，事件如何影响商机、线索、预测或报告。
6. 缓存口径：缓存服务什么页面或分析，刷新规则是什么。
7. 证据链规则：关键结论、金额、日期、客户状态、政策依据和风险原因来自哪里。
8. API 契约：请求、响应、分页、过滤、排序、错误信息和审计记录。
9. 页面闭环：保存、删除、测试、运行、导出、确认、撤销、回滚之后，用户如何看到结果。

如果缺少契约，不允许直接在代码里硬编码字段名、状态值、页面选项、接口路径、提示词兜底逻辑或业务判断。Agent 必须先输出“契约缺口清单 + 最小补齐方案”，不能假装已经理解。

只有用户明确要求快速原型时，才允许短期 mock。mock 必须标记为临时能力，并写清楚验收点、失效条件和后续补齐路径。

## 3. 本体数据库优先

本项目核心不是表，而是市场与销售增长本体。当前至少包含：

1. 外部情报：标讯、政策、市场动态、竞对动态、行业知识。
2. 线索：情报转化出的可行动机会，包括来源、匹配原因、金额、区域、客户、产品、负责人和状态。
3. 商机：销售确认后的机会，包含阶段、金额、预计签单时间、风险、下一步动作和证据。
4. 签单预测与回款预测：来自销售预测、历史事实、商机事件和 Agent 跟进，不允许无来源覆盖。
5. 客户、组织、人员、区域、产品、业务线。
6. 销售活动：拜访、交流、方案、投标、POC、合同、回款跟进等。
7. 经营事件与情报事件：解释事实、预测和风险为什么变化。
8. 日报、周报、月度洞察和管理复盘。

事实、关系、事件、缓存都必须服务于本体语义，不能为了页面方便随便新造一套口径。

签单、收入、回款、成本、销售活动、标讯原文、政策原文、竞对事件是事实或来源材料。经营事件和情报事件解释变化。缓存只用于提速和汇总，不能作为原因解释来源。

## 4. 市场洞察 Agent 规则

市场洞察不是“采集列表”。每个 Agent 都必须完成从数据到判断再到行动建议的闭环。

标讯雷达 Agent 必须至少支持：

1. 根据管理中心关键词和行业口径过滤高相关标讯。
2. 抽取项目名称、采购人、预算/中标金额、地区、行业、时间、采购阶段、来源和原文链接。
3. 给出相关性评分、匹配关键词、匹配原因、风险点和建议动作。
4. 按周、月分析行业分布、区域分布、关键词触发、金额分布、重点客户和异常变化。
5. 支持线索转入商机中心，并保留来源证据。

政策与市场跟踪 Agent 必须至少支持：

1. 围绕公安、政数、大数据、电力、运营商、地址、地图、空间智能、Agent、数据治理等高相关领域采集。
2. 以自然年、季度、月份进行政策归集和趋势分析。
3. 输出市场导向、影响对象、潜在机会、风险限制和建议动作。
4. 明确引用政策来源、发布日期、发布单位和原文链接。

竞对监控 Agent 必须至少支持：

1. 监控竞对中标、客户案例、产品发布、区域动作、合作伙伴、招聘方向和公开宣传。
2. 竞对名单必须可配置，当前重点包括但不限于武大吉奥、京东舆图、海致等。
3. 输出“发生了什么、影响哪个市场、可能抢什么机会、我们该做什么”。
4. 不允许只抓新闻标题而没有业务判断。

行业动态知识补充必须服务售前和市场动作，不允许采集泛 AI 噪声。采集内容必须能沉淀为方案素材、话术素材、竞品判断或市场趋势。

## 5. 采集源与反爬规则

任何采集源必须区分状态：

1. 已接入：真实参与运行，有解析规则，有日志，有失败反馈。
2. 待配置：后台可见但不参与运行，必须说明缺少什么。
3. 候选源：仅供管理员评估，不能伪装成已接入。
4. 失效源：连续失败或结构变化，必须提示原因。
5. 禁用源：管理员关闭后不得参与采集。

采集源配置必须包含来源名称、来源类型、URL 或 API 地址、数据类别、采集策略、反爬策略、频率限制、失败退避、字段映射、解析规则、健康状态和最近运行日志。

如果后台允许新增或启用采集源，真实爬虫必须消费该配置。不能出现“后台能启用，但运行链路完全忽略”的假功能。

涉及政府采购、政策法规、地方官网和公开网站采集时，必须遵守低频请求、来源白名单、失败退避、超时控制、去重、robots/站点规则评估和人工可关闭机制。不得使用攻击性绕过、撞库、暴力请求或破坏性手段。

浏览器渲染采集、验证码处理、登录态采集、结构化 API 采集必须明确能力边界。尚未实现时，页面必须标为“待接入”。

## 6. 管理中心闭环规则

管理中心所有配置必须真实影响后端行为或生成真实产物。

以下功能不得只保存不生效：

1. 采集关键词：必须影响召回、过滤、评分或分析口径，并说明作用位置。
2. 采集源：必须被对应爬虫读取，运行日志中能看到来源。
3. 结构化标讯配置：必须被标讯采集链路使用，失败时给出明确错误。
4. LLM 配置：必须被所有模型调用统一读取，不能只有测试连接生效。
5. Prompt 模板：必须被对应 Agent、分析任务或报告任务使用。
6. API Key：必须能用于受保护接口鉴权，支持启用、禁用、删除和审计。
7. 用户与权限：必须影响登录、页面访问和接口访问。
8. 调度配置：必须被真实调度器消费，能看到下次运行时间和历史日志。
9. 钉钉配置：必须能测试、保存、发送真实通知，并记录失败原因。

如果功能尚未打通，必须在页面上清楚标注“未接入”“待配置”或“原型”，不能让用户误以为已经生效。

## 7. 面向最终用户

所有页面文字必须面向市场负责人、销售管理者和业务管理者，而不是面向开发人员。

禁止把以下内容直接写到前端页面：

1. 内部讨论、用户提醒、临时说明。
2. Prompt 设计思路。
3. 测试话术、调试信息和默认账号密码。
4. 数据库表名、技术字段和接口路径，除非页面就是系统管理员配置页。
5. “这是为了演示”“后续再说”等没有产品价值的说明。

页面要少废话，多展示可操作对象、业务依据、证据、影响范围、判断结论和下一步建议。错误信息必须告诉用户问题在哪里、能不能重试、需要谁处理。

## 8. AI 工作台规则

AI 工作台不是数据库查询页面，也不是固定意图路由器。模型负责理解问题、拆解任务、选择工具、基于证据组织结论；平台负责本体注入、工具边界、查询执行、证据链、权限、安全和审计。

每次涉及经营数据、市场情报或商机判断的问答，必须向模型提供：

1. 市场与销售增长本体知识。
2. 维度词典：人员、组织、业务线、区域、客户、产品、竞对、关键词。
3. 关系图：情报到线索、线索到商机、商机到预测、预测到复盘。
4. 指标口径：线索数、相关性、金额、阶段、签单预测、回款预测、转化率、风险等级。
5. 业务口语映射。
6. 受控取数工具说明。
7. 反虚构规则和证据引用规则。

平台不应强行替模型判断唯一业务意图。涉及数据库和外部情报的问题必须使用受控工具取数，模型不允许绕过工具编造数据。

## 9. 取数与证据链

经营数据和市场情报回答必须来自受控工具返回的 Evidence Record。Evidence Record 至少包含：

1. `evidence_id`
2. `source`
3. `source_type`
4. `query_summary`
5. `record_ids`
6. `data`
7. `confidence`
8. `data_quality_flags`
9. `collected_at` 或 `event_time`
10. 原文链接、截图、文件或接口响应摘要

关键数字、客户状态、金额、时间、风险原因、政策依据和竞对判断必须能追溯到证据。没有证据的数据不能说成确定结论；推测必须标注为推测。

查询工具必须支持按人员、组织、业务线、区域、客户、产品、时间、状态、来源、关键词、竞对和行业组合取数。

## 10. 经营事件与商机闭环

不允许直接改关键数字但没有原因。

商机阶段、预计签单时间、预测金额、回款计划、风险等级、重点客户状态、线索转化结论等变化，必须形成经营事件或关联已有事件。

批量导入、自然语言输入、文件输入、销售对话和外部情报采集，都应先转成可确认的事件或线索，再驱动商机、预测和复盘变化。

低风险事件可以批量确认。高风险事件必须突出展示影响范围、原值新值、原因、责任人、证据和下游影响。

## 11. 配置管理

`.env` 只放启动级配置，例如数据库地址、服务端口、日志级别和默认启动开关。

运行时可变配置必须在管理中心维护，包括：

1. LLM 服务地址、模型名称、API Key 和模型路由。
2. OCR、截图、文档解析服务配置。
3. 外部 API、结构化数据接口和外部数据源凭证。
4. 采集源、关键词、反爬策略、调度策略和阈值。
5. Prompt、Skill、Tool、Agent 配置。
6. 钉钉、API Key、通知规则和审计规则。

配置页面只校验格式、连通性和协议兼容性，不替用户做业务策略判断。不得提交真实密钥，不得把用户本机临时密钥、模型中转站地址或内部服务凭证写入仓库文档。

## 12. 功能必须闭环

禁止做“看起来有”的假功能。

上传、保存、删除、测试连接、运行 Agent、生成看板、确认事件、导出文件、回滚版本、启用源、禁用源、调度任务、发送通知和转入商机，都必须真实影响后端状态或生成真实产物。

每个闭环功能至少要明确：

1. 用户输入。
2. 后端动作。
3. 数据落点。
4. 状态变化。
5. 审计记录或运行日志。
6. 成功反馈。
7. 失败反馈。
8. 如何再次查看结果。

如果只是原型，必须明确标记为原型，并说明验收点和后续补齐路径。

## 13. 数据生成规则

模拟数据必须符合本体设计，不能随机堆垃圾。

数据要能支持真实业务问题验证，例如：

1. 某业务线本月/本年新增多少高相关线索。
2. 某区域有哪些重点标讯和政策机会。
3. 某销售名下商机的签单预测和回款预测。
4. 某客户相关标讯是否已经转商机。
5. 某商机经历了哪些经营事件。
6. 回款逾期原因和影响。
7. 标讯金额抽取是否准确。
8. 政策趋势是否能解释市场动作。
9. 竞对动作是否影响重点客户。
10. 日报周报是否能回答管理者“大家在做什么、下一步盯什么”。

生成数据必须有固定 seed、规模统计、场景覆盖、插入顺序、校验脚本和可复算缓存。旧模拟数据重建时必须明确清理旧数据，避免新旧口径混杂。

## 14. 页面设计规则

页面职责必须清晰：

1. 日报周报：管理者复盘和行动提醒。
2. 市场洞察：外部情报分析和市场机会判断。
3. 商机中心：销售跟进、Agent 追问、签单预测和回款预测。
4. 管理中心：配置、权限、运行监控和系统保护。

页面不能把数据治理、数据修正、数据接入、分析结论和管理配置拆成一堆重复工作台。每个页面必须有清楚的主任务、主对象和主动作。

市场洞察页面不能只展示原始列表，必须有高相关关注、行业分布、关键词分布、金额分布、趋势变化和建议动作。商机中心不能强融合情报采集逻辑，商机追问有自己的商机机制，Agent 只跟进进度和风险。

## 15. 部署与访问控制

当前可以优先支持内网单账号部署，但代码结构不能写死成无权限系统。

如果未来暴露到公网、域名、IP 或端口可访问，必须默认拒绝未授权访问。除健康检查、登录页、静态资源和明确公开信息外，以下能力都必须要求有效登录会话或服务端凭证：

1. 业务 API。
2. 管理后台 API。
3. Agent 和 LLM 调用。
4. 数据导入、导出和回测。
5. 模型、外部工具、采集源和密钥配置。
6. 审计、调度和任务队列。
7. API Key 管理。

不能因为“现在是内网”就在代码结构上写死无权限设计。

## 16. Git 与交付

每次修改必须保持工作区清晰，不能留下无用文件、过期文档、临时脚本、调试输出或半成品文档。

提交或交付前必须说明：

1. 改了哪些文件。
2. 为什么改。
3. 影响哪些页面、接口、数据、Agent 或测试。
4. 运行了哪些测试。
5. 当前限制和下一步建议。
6. 是否推送到 GitHub。

不得私自重置或覆盖用户未确认的改动。发现用户或其他 Agent 已修改同一文件时，必须先理解改动意图，再合并处理。

## 17. 测试与验收

测试不能只验证接口返回 200。必须验证业务问题是否答对、范围是否正确、证据是否存在、数字是否合理、页面是否闭环。

高频验收问题必须固化为测试用例，至少覆盖：

1. 本周有哪些高相关标讯值得关注，为什么。
2. 标讯金额是否从标题、正文、预算字段或中标字段中正确抽取。
3. 政策类数据是否覆盖指定自然年，是否能解释市场导向。
4. 市场线索是否围绕公安、政数、大数据、电力、运营商、地址、地图、空间和 Agent 相关主题。
5. 竞对本月有哪些动作，影响哪些客户或区域。
6. 某条线索如何转入商机，证据是否保留。
7. 某销售商机预测变化的原因是什么。
8. 回款预测和逾期风险是否有事件支撑。
9. 日报周报是否能回答管理者“团队在做什么、风险在哪里、下一步盯什么”。
10. 管理中心配置保存后是否真实影响运行链路。

每个测试必须写清楚：输入是什么，期望答案包含什么，哪些回答算失败。涉及页面交互时必须进行浏览器验收；涉及 Agent 和模型输出时必须验证 FakeLLM 或真实模型关键路径；涉及采集时必须验证运行日志、入库数据和分析结果。

## 18. Code Review 红线

审查代码时必须优先检查以下问题：

1. 是否新增硬编码字段、枚举、状态值、角色名、来源类型或业务口径。
2. 是否重复创建主数据模型、状态机、权限判断或取数逻辑。
3. 是否把缓存当事实来源。
4. 是否把前端文案当业务规则。
5. 是否把平台规则写死，限制模型自主理解和受控取数。
6. 是否出现页面能看但后端没有闭环的假功能。
7. 是否后台配置保存了但运行链路不读取。
8. 是否有接口契约漂移，前端靠猜字段工作。
9. 是否有模型超时、工具空数据、采集失败、文件格式不支持、浏览器刷新等异常路径未处理。
10. 是否有测试只验证成功状态，不验证业务答案。
11. 是否把用户提醒、内部讨论或临时说明写进产品页面。

发现红线问题，应先报告风险和建议修复路径。必要时直接阻止合并。

## 19. 交付前自检清单

交付前必须逐项确认：

1. 是否读过本文件和当前 `README.md`。
2. 是否没有违背项目模块边界。
3. 是否没有新增硬编码业务口径。
4. 是否功能真实闭环。
5. 是否关键结论有证据来源。
6. 是否后台配置真实影响运行链路。
7. 是否页面文案面向最终用户。
8. 是否异常路径有用户可理解的反馈。
9. 是否补充或更新测试。
10. 是否运行相关测试，必要时运行全量测试。
11. 是否做过浏览器验收。
12. 是否清理无用文件和临时产物。
13. 是否说明修改内容、测试结果、当前限制和下一步建议。

如果任何一项不能确认，不能假装完成。必须明确说明未完成原因、风险和下一步处理方案。
